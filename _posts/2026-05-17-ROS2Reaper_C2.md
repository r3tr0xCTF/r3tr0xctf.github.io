## tl;dr

ROS2Reaper is a modular offensive framework targeting ROS2/DDS, ICS/OT, and AWS IoT
Greengrass environments. The piece I want to talk about here is the **C2 layer** — the
part that makes a swarm of robots, a Modbus PLC, and a Greengrass core all controllable
from one operator console without pretending they're the same thing.

The interesting design problem wasn't "how do I beacon." It was: **the targets are
already talking on their own pub/sub fabric, and that fabric was designed under the
assumption that everyone on it is friendly.** A good C2 here doesn't impose a new
channel — it impersonates a legitimate participant on the existing one.

Three things I want to walk through:

1. **What ROS2/DDS actually gives you for free as an attacker** (a lot)
2. **Architecture decisions** I had to make for the C2 to work across ROS2, ICS, and Greengrass without becoming three tools
3. **Attack scenarios** the C2 enables that aren't possible with off-the-shelf pentest gear

If you saw the talk, this is the part where I would have kept going if I had another hour.

## 1. The fabric eats the C2 problem for you

Most C2 frameworks spend enormous engineering effort on the boring parts: transport,
discovery, peer enumeration, NAT traversal, message reliability, multicast for one-to-many,
encryption negotiation. ROS2/DDS gives you **all of that for free**, because that's
what DDS *is*. RTPS — the wire protocol underneath ROS2 — is a pub/sub system with:

- Multicast peer discovery on a well-known port (`7400` + offsets, by default)
- Built-in topic-based addressing (you don't need to know IPs; you subscribe to a name)
- QoS-driven reliability (RELIABLE topics will redeliver, BEST_EFFORT will fire-and-forget)
- Optional, often *unenabled*, DDS-Security authentication and access control

In a default-config ROS2 deployment — which is *most* of them, in my experience auditing
this stuff — a new participant on the same domain is indistinguishable from a legitimate
one. There is no handshake to fail, no certificate to forge, no JWT to steal. You join
the domain. You're in.

For the C2 designer, this flips the usual cost equation. Implants don't need a callback
channel — they're already on one. The operator doesn't need to enumerate hosts — the
discovery protocol announces them. Tasking doesn't need to be encrypted to avoid IDS —
the entire fabric is binary RTPS that most IDS sensors don't parse.

This is the design insight the whole C2 is built around: **don't build a covert
channel; ride the overt one.**

## 2. Architecture: one operator, three fabrics

The challenge ROS2Reaper had to solve is that real ICS/OT environments aren't *just*
ROS2. The lab I built for the talk has, on the same `10.52.32.0/24` macvlan:

- A multi-robot ROS2 simulation (TurtleBots in Gazebo, talking DDS)
- A Modbus/TCP PLC and a DNP3 outstation
- An OPC UA server fronting the PLC tags
- An MQTT broker bridging to EtherCAT-flavored telemetry
- An AWS IoT Greengrass core, running Lambdas, with Shadow + Jobs

An operator doing a real engagement against a logistics facility, a research robotics
lab, or a smart-factory pilot is going to see *all of these on the same VLAN*. A C2 that
only speaks DDS is a toy. A C2 that tries to speak all of them natively becomes a
swiss-army monstrosity.

The compromise I landed on:

```
                   ┌────────────────────────────┐
                   │   ros2reaper console (op)  │
                   └─────────────┬──────────────┘
                                 │
                  ┌──────────────┴──────────────┐
                  │   tasking + result bus      │   ← spoken in DDS,
                  │   (a DDS domain we own)     │     because we already
                  └──────┬─────┬──────┬─────────┘     have the stack
                         │     │      │
                   ┌─────┴┐ ┌──┴───┐ ┌┴────────┐
                   │ ros2 │ │ ics  │ │ gg-iot  │
                   │ mods │ │ brdg │ │ bridges │
                   └──────┘ └──────┘ └─────────┘
                       │       │          │
                  (RTPS on   (Modbus,   (AWS IoT
                   target     DNP3,     MQTT/TLS,
                   domain)    OPC UA,   Shadow,
                              MQTT)     Jobs)
```

The **tasking and result bus is itself a DDS domain.** Operator console publishes tasks
on a topic like `r2r/task/<module>/<target>`; modules subscribe to the slice they care
about and publish results back on `r2r/result/...`. Each protocol-specific module
(Modbus bridge, OPC UA bridge, Greengrass bridge, etc.) is a translator: DDS in on the
operator side, native protocol out on the target side.

Why this is worth doing, instead of e.g. a plain TCP control channel:

- **The operator gets one console, one query language, one transport** regardless of what
  they're actually attacking. Reuse of DDS for tasking means no separate auth, no separate
  serialization, no separate session management.
- **On a compromised host inside the OT network, the C2 traffic looks like ROS2 traffic**,
  because it is. RTPS on UDP, multicast discovery, the works. If a defender's already
  given up on inspecting DDS — and most have — your C2 inherits that blind spot.
- **Modules are independently shippable.** The Greengrass bridge doesn't know the Modbus
  bridge exists. They just both subscribe to the operator's tasking bus.

The thing I want to flag, because someone will ask: this is *not* DDS-Security-aware.
If a target environment has DDS-Security configured with mutual auth and access control,
the C2's "free" entry into the target domain disappears and you're back to stealing keys.
In practice, in 2026, I have yet to encounter a deployed ROS2 stack with DDS-Security
fully on. Your mileage may vary; please prove me wrong.

## 3. Attack scenarios the C2 makes cheap

These are the demos. The talk shows two of them live; the rest are in the README's
31-scenario matrix with CVSS and MITRE ATT&CK ICS mappings.

### 3.1 Swarm `cmd_vel` injection

The cleanest demo, and the one that gets the gasp in the room. ROS2's standard motion
topic for differential-drive robots is `/cmd_vel`, carrying `geometry_msgs/Twist`. Any
participant in the domain can publish to it. There is no notion of "the legitimate
controller" at the protocol layer.

From the operator console:

```
r2r> use ros2/swarm_cmdvel
r2r(swarm_cmdvel)> set domain 0
r2r(swarm_cmdvel)> set pattern circle
r2r(swarm_cmdvel)> run
```

The module enumerates participants via discovery, identifies anything advertising a
`cmd_vel`-shaped publication, and starts publishing `Twist` messages at a higher rate
than the legitimate planner. Because most robot stacks take the *latest* `cmd_vel`, the
attacker wins by being louder. In the lab, three TurtleBots in Gazebo start describing
a synchronized circle while their behavior trees scream into the void.

In a logistics warehouse or a research lab, the equivalent is AGVs that no longer follow
their planned paths. The point of the demo is not the circle — it's that *the protocol
provides no mechanism to reject this*. You'd need a layer above ROS2 to enforce
"only the planner can write `cmd_vel`," and that layer almost never exists.

### 3.2 Greengrass Shadow + Jobs abuse → fleet OTA

This is the scenario that lands at CVSS 10.0, and it's where the C2 design pays off
the most.

AWS IoT Greengrass v2 cores ship with three abusable primitives, if you can talk to the
device-side MQTT broker (or, better, steal the X.509 cert off a compromised core):

1. **Device Shadows** — desired/reported state docs. If you can write `desired`, the
   core's components will obediently try to converge to it.
2. **IoT Jobs** — fleet-wide tasking, with rollout controls. A Job document can
   reference an arbitrary S3 artifact and a target group of things.
3. **Component deployments** — Greengrass's mechanism for shipping new code to cores.

The C2's Greengrass bridge subscribes to operator tasking and translates it into AWS
IoT MQTT messages on the relevant reserved topics (`$aws/things/.../shadow/update`,
`$aws/things/.../jobs/...`). The operator types:

```
r2r> use aws/gg_jobs_ota
r2r(gg_jobs_ota)> set thing_group prod-edge-fleet
r2r(gg_jobs_ota)> set artifact s3://...
r2r(gg_jobs_ota)> run
```

…and a fleet OTA fans out to every core in the group. The cores authenticate the
operation using their *own* X.509 certs against AWS IoT — which is correct behavior
from AWS's perspective. The compromise is at the issuing edge, not in the cloud.

What makes this a C2 problem and not just a Greengrass abuse script is the targeting.
The DDS tasking bus lets the operator say "this Job, against the things I identified
via the Shadow fingerprinting module thirty seconds ago" without ever leaving the
console. Bridges talk to bridges via the result bus, so fingerprint output from
`gg_fingerprint` is directly consumable as input to `gg_jobs_ota`. That composition
is what makes the framework feel like a C2 instead of a folder of scripts.

### 3.3 OPC UA → DDS lateral movement

Underdiscussed scenario, but the one I think will age the best as more vendors bolt
ROS2 onto industrial controllers. The OPC UA bridge enumerates an OPC UA server's
address space, looks for nodes that mirror process variables also exposed on the DDS
side, and lets the operator drive both simultaneously — either to mask an attack on
one side by faking telemetry on the other, or to compare the two and find drift.

Concrete example: a Modbus coil controls a valve. OPC UA exposes the coil's state to
the HMI. ROS2 exposes the *desired* state to a robotic arm coordinating with the
valve. Operator writes the coil via Modbus, suppresses the OPC UA update so the HMI
shows the old state, and publishes a fake "valve closed" message on the DDS topic the
arm reads. The arm proceeds as if the valve were closed. The valve is not closed.

The C2's job here is to keep the three lies synchronized. Doing this from three
separate scripts is a coordination nightmare. Doing it from one console where every
action is a tasking message on a shared bus is — and I cannot stress this enough — *easy.*

## 4. What I'd build differently next time

A few honest notes, since this is a companion piece and not marketing:

- **The DDS tasking bus is a single point of failure for the operator.** If you lose
  the operator host, every in-flight module gets orphaned. I'd add an operator-side
  durability layer next.
- **No DDS-Security story for the C2's own bus.** Right now anyone who joins the
  operator domain can see tasking. In a red team / blue team engagement that's fine;
  in a multi-operator scenario it isn't. This is on the Phase 4 list.
- **Cisco IR/IW modules are not in this release.** Deferred to Phase 4 because the
  Cisco IOx surface is its own animal and I didn't want to ship it half-done.
- **Detection content is sparse.** I owe defenders a Suricata ruleset for the RTPS
  patterns this thing generates. That's the next blog post.
